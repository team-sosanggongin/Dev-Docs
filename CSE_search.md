# 클라이언트 사이드 암호화 (CSE) 조사 문서

 **목적**: 모바일 앱(Flutter/Dart)에서 민감 데이터를 클라이언트 사이드에서 암호화하여
 서버 운영자도 데이터에 접근할 수 없는 구조를 위한 방법론 조사

**방향**: 결정 전 단계의 조사 문서. 다향한 방법을 기술하며 결정은하지 않음

**핵심 조건**
 - 조건1: 서버에는 암호화 키가 절대 올라가지 않아야 함
 - 조건2: 기기 분실 시에도 복구 가능해야 함
 

---

## 1. 개요

### CSE란

데이터를 서버로 전송하기 전에 클라이언트(앱) 단에서 암호화하는 방식.
서버에는 암호화된 데이터(암호문)만 저장되며, 복호화에 필요한 키는 서버에 존재하지 않음.

```
일반 구조:
[평문 데이터] → [클라이언트에서 암호화] → [암호문만 서버로 전송/저장]
                        ↑
               [키는 클라이언트에만 존재 → 서버 운영자 접근 불가]
```

###  용어 설명

| 용어 | 설명 |
|---|---|
| **DEK** (Data Encryption Key) | 실제 데이터를 암호화하는 키 |
| **KEK** (Key Encryption Key) | DEK를 암호화하는 마스터 키(열쇠(DEK)를 또다른 열쇠로 잠구는방식) |
| **KDF** (Key Derivation Function) | 사용자가 입력한 비밀번호는 짧고 예측 가능해서 암호화 키로 바로 쓰기엔 취약 <br> 이 비밀번호를 가공해서 복잡한 암호화 키로 변환해주는 함수 |
| **FFI** (Foreign Function Interface) | Dart에서 직접 구현하기 어려운 기능이 필요할때  네이티브 라이브러리를 호출하는 인터페이스|

## 2. 키 관리 아키텍처

CSE에서 가장 중요한 설계 결정은 키를 어디에, 어떻게 보관하느냐이다.

### 2-1. Password-derived Key (비밀번호 파생 키)

사용자 비밀번호를 키 파생 함수에 통과시켜 암호화 키를 생성하는 방식.
비밀번호 자체는 절대 서버로 전송되지 않음.

```
[비밀번호 입력]
      ↓
키 파생 함수로 잠금 해제 키 생성
(메모리에만 잠깐 존재, 저장되지 않음)
      ↓
잠금 해제 키로 데이터 암호화 키를 잠금
      ↓
잠긴 데이터 암호화 키 → 서버 저장 (잠금 해제 키 없이는 쓸모없음)
데이터 암호화 키 → 실제 데이터 암호화에 사용
 
서버에 저장되는 것: 잠긴 데이터 암호화 키 + salt (키 파생에 필요한 랜덤값)
서버에 없는 것: 비밀번호, 잠금 해제 키, 데이터 암호화 키 (평문)
```

**서버 저장 정보**: 서버에는 잠긴 데이터 암호화 키와 salt만 저장. 비밀번호와 실제 키는 서버에 없음
**기기 분실 복구**:  비밀번호만 알면 어느 기기에서든 잠금 해제 키를 재생성해서 복구 가능

**장점**
- 비밀번호만 알면 어느 기기에서든 복구 가능
- Dart 단독 구현 가능

**단점**
- 보안이 비밀번호 강도에 완전히 종속, 약한 비밀번호 = 낮은 보안
- 비밀번호 분실 = 복구 불가 (서버도 비밀번호를 모름)
- 비밀번호 변경 시 데이터 암호화 키 재암호화 필요

**특징**
- 키 파생 함수 연산 자체는 라이브러리가 제공하므로 직접 수학 구현 불필요
- 분리 구조 설계, 잠긴 데이터 암호화 키 저장/복호화 플로우, salt 관리 구현

---

### 2-2. Shamir's Secret Sharing (SSS, 비밀 분산)

키를 N개 조각으로 분산하고, K개 이상의 조각이 모여야만 원래 키를 복구할 수 있는 방식.

```
예: (2, 3) threshold — 3개 조각 중 2개만 있으면 복구 가능

[KEK] → SSS → [조각1: 서버] [조각2: 이메일] [조각3: 기기]

복구 시: 조각 2개 이상 조합 → [KEK 복구]
공격자: 조각 1개만 탈취 →  복구 불가
```

**서버 저장 정보**: 조각 하나만으로는 키 복구 불가. 서버가 조각 하나를 갖더라도 복호화 불가
**기기 분실 복구**: 조각이 분산 보관되므로 기기 분실 시 다른 조각으로 복구 가능

**장점**
- 조각 하나가 탈취돼도 키 복구 불가
- 조각을 이메일, 서버, 기기 등 다양한 곳에 자유롭게 분산 가능
- 몇 개 조각이 있어야 복구되는지 설정으로 조절 가능
  

**단점**
- 조각 배포 채널 자체의 보안도 별도로 관리해야 함
- 일정개수 미달 시(조각 분실) 복구 불가
- 조각이 변조되지 않았는지 검증하는 로직을 별도로 구현해야 함

**특징**
- 조각 배포/수집/검증 플로우 설계가 복잡함.


**참고자료**: [Wikipedia - Shamir's Secret Sharing](https://en.wikipedia.org/wiki/Shamir%27s_secret_sharing)

### 2-3. SRP (Secure Remote Password)

비밀번호 기반의 상호 인증 프로토콜
클라이언트와 서버가 수학적 교환을 통해 세션 키를 도출하며, 이 과정에서 비밀번호는 네트워크를 통해 절대 전송되지 않음.
서버에는 비밀번호 검증자(verifier)만 저장되며, 이것만으로는 비밀번호나 키를 복원할 수 없음.

```
[앱]                                    [서버]
비밀번호 보유                            검증자만 저장 (비밀번호 모름)
    ↓
공개값 A 전송 ──────────────────────→
              ←────────────────────── 공개값 B + salt
    ↓
비밀번호로 세션 키 계산                   비밀번호 없이 세션 키 계산
(비밀번호는 앱 밖으로 절대 안 나감)
    ↓
세션 키 → 데이터 암호화에 활용
```

**서버 저장 정보**: 비밀번호와 암호화 키가 서버로 전송되지 않음. 서버는 검증자만 보유
**기기 분실 복구**: 비밀번호만 알면 어느 기기에서든 동일한 키 도출 가능

**장점**
- 비밀번호가 네트워크를 통해 절대 전송되지 않음 (평문은 물론 해시값도 아님)
- 서버 DB가 탈취되어도 비밀번호 복원 불가 (검증자만 저장)
- 클라이언트와 서버가 서로의 신원을 상호 인증 (중간자 공격 방어)
- Dart 전용 패키지 존재 (`dsrp`) → 단독 구현 가능

**단점**
- Password-derived Key보다 구현 복잡도가 높음
- SRP 자체는 인증 프로토콜이므로, 인증 후 데이터 암호화를 위한 키 관리 구조는 별도 설계 필요
- 비밀번호 분실 시 복구 불가 (서버도 비밀번호를 모름)

**특징**
SRP 프로토콜 자체는 `dsrp` 패키지가 처리하므로 직접 수학적 구현 불필요
백엔드에도 SRP 서버 측 구현 필요 검증자 저장, 라운드트립 처리




### 2-4. ZKP 기반 (Zero-Knowledge Proof, 영지식 증명)

키 소유 사실을 증명, 키 자체는 단 한 번도 공개하지 않는 방식.
인증 과정에서 서버가 어떤 비밀 정보도 수신하지 않음.

```
일반 인증: [사용자] → 비밀번호(또는 키) 전송 → [서버가 검증]
ZKP 인증:  [사용자] → 나는 이 키를 갖고 있다는 증명값 → [서버가 검증]
                         (키 내용은 서버가 알 수 없음)
```

**서버 저장 정보**:  키 자체가 서버에 전송되지 않음. 증명값만 전달
**기기 분실 복구**:  증명 기반이므로 기기와 무관하게 복구 가능 (비밀 보유 시)

**장점**
- 인증 과정에서 서버가 비밀 정보를 수신하지 않으므로 서버 탈취 시에도 키 유출 없음

**단점**
- 구현 난이도 어렵다고함
- 증명 생성 연산 비용이 매우 큼. 모바일 환경에서 응답 지연 발생 가능
- Dart 전용 라이브러리 없음 → FFI 바인딩 필요

**특징**
ZKP는 암호학 분야에서도 고급 주제에 속하며, 구현 실수가 보안 전체를 무너뜨릴 수 있는 위험이 높음.


**레퍼런스**: [Wikipedia - Zero-knowledge proof](https://en.wikipedia.org/wiki/Zero-knowledge_proof)

### 2-5. 니모닉(Mnemonic) 기반 키
 
사용자가 기억하거나 종이에 적어두는 12~24개의 단어 조합(니모닉)으로 키를 파생하는 방식.
암호화폐 지갑에서 표준으로 사용 중.
서버에는 어떤 키 관련 정보도 저장하지 않으며, 니모닉만 알면 어느 기기에서든 동일한 키를 복원할 수 있음.
 
```
[12~24개 단어] → [KDF] → [마스터 키]
                                      ↓
                              [DEK 파생 → 데이터 암호화]
서버: 암호문만 저장
```
 
**서버 저장 정보**: 암호문만 
**기기 분실 복구**: 니모닉만 있으면 어느 기기에서든 복구 가능
 
**장점**
- 서버에 키 관련 정보 일절 없음
- 니모닉만 보관하면 완전한 복구 가능 (기기 독립적)
- 암호화폐 지갑 생태계에서 검증
- 
**단점**
- 니모닉 분실 = 복구 불가 (서버에 없음)
- 니모닉을 디지털로 저장하면 보안 위협 (클라우드 메모, 캡처 등)
- UX가 불친절함 (일반 사용자에게 12개 단어 관리는 낯선 경험)

  
**특징**
Dart 전용 라이브러리 없음 → FFI 바인딩 또는 직접 구현 필요.
 
 
**레퍼런스**: [BIP-39 표준](https://github.com/bitcoin/bips/blob/master/bip-0039.mediawiki)

### 2-6. Secure Enclave 단독 (기기 바인딩)
 
키를 기기의 하드웨어 보안 모듈(iOS: Secure Enclave, Android: StrongBox)에 완전히 묶는 방식.
키가 하드웨어 금고 안에 격리되어 있어 OS가 루팅되어도 꺼낼 수 없음.
 
```
[키 생성]
    ↓
Secure Enclave / StrongBox 내부에 저장
(하드웨어 외부로 절대 추출 불가)
    ↓
생체인증(지문, Face ID) → 하드웨어 금고 잠금 해제 → 데이터 복호화
 
서버: 암호문만 저장
```
 
**서버 저장 정보**: 서버에 키 관련 정보 일절 없음. 암호문만 저장
**기기 분실 복구**: 키가 기기 하드웨어에 종속되므로 기기 분실 시 복구 불가
 
**장점**
- 키가 하드웨어 외부로 절대 추출 불가. OS 루팅시에도 보호
- 생체인증과 하드웨어 수준으로 연동 가능

  
**단점**
- 기기 분실, 초기화, 파손 시 키 소멸 → 데이터 영구 복구 불가 (조건2 미충족)
- 멀티 기기 지원 불가 (키가 특정 기기 하드웨어에 종속)

  
**특징**: 낮음
Android Keystore / iOS Keychain API가 추상화를 제공하므로 직접 암호화 로직 구현 최소화.
Flutter 패키지: `flutter_secure_storage`
 
> **조건2를 충족하려면**: 다른 방식(SSS, 니모닉 등)과 조합 필요. 3번 조합 패턴 참고.
 
**레퍼런스**: [Android Keystore System](https://developer.android.com/privacy-and-security/keystore), [Apple Secure Enclave](https://support.apple.com/guide/security/secure-enclave-sec59b0b31ff/web)

## 3. 조합 패턴
 
각 아키텍처를 단독으로 쓰면 한계가 있어서, 메인 방식 + 복구 수단을 조합해서 설계를 많이 한다고 한다.
 
### 단독 방식의 한계
 
| 방식 | 단독 사용 시 한계 |
|---|---|
| Secure Enclave 단독 | 기기 분실 시 복구 불가 |
| 비밀번호 파생 키 단독 | 비밀번호 분실 시 복구 불가 |
| SSS 단독 | 데이터에 접근할 때마다 조각을 수집해야 해서 평소 사용 UX 불가 |
| 니모닉 단독 | 12~24단어 관리가 일반 사용자에게 낯섦 |
 
> **SSS를 메인으로 쓰기 어려운 이유**
> 데이터에 접근할 때마다 조각 K개를 모아야 하는데, 조각이 서버/이메일/기기에 분산되어 있으면
> 매번 이 과정이 필요해서 현실적으로 사용 불가능함.
> 그래서 SSS는 주로 **최초 키 생성 시 또는 분실 복구 시**에만 사용하고,
> 평소엔 다른 방식으로 키에 접근하는 구조로 설계함.
 
---
 
### 3-1. Secure Enclave + SSS
 
평소엔 기기 하드웨어(Secure Enclave)로 키를 보호하고, 기기 분실 시 SSS 조각으로 복구하는 방식.
카카오 로그인처럼 별도 비밀번호가 없는 환경에 적합.
 
**최초 가입 시**
```
카카오 로그인 (서비스 회원 확인)
        ↓
Secure Enclave에서 키 생성
        ↓
이 키를 SSS로 3개 조각으로 분산
        ↓
조각1 → Secure Enclave 보관 (평소 사용)
조각2 → 서버에 저장
조각3 → 사용자 이메일로 전송
```
 
**평소 사용**
```
카카오 로그인 (서비스 회원 확인)
        ↓
생체인증 (기기 주인 확인)
        ↓
Secure Enclave에서 키 꺼내 데이터 복호화
```
 
**기기 분실 시 복구**
```
새 기기에서 카카오 로그인
        ↓
서버 조각2 + 이메일 조각3 수집 (2개면 복구 가능)
        ↓
키 복원 → 새 기기 Secure Enclave에 저장
        ↓
복구 완료
```
 
**장점**
- 평소엔 생체인증으로 자연스럽고 빠른 UX
- 기기 분실 시 SSS로 복구 가능
- 서버는 조각 하나만 갖고 있어서 단독 복호화 불가
- 별도 비밀번호 없이 카카오 로그인만으로 운용 가능
  
**단점**
- 이메일 조각3 분실 시 복구 불가 (기기 조각1도 기기와 함께 소멸)
- 이메일 계정 해킹 시 서버 조각2 + 이메일 조각3 = 2개가 공격자 손에 들어갈 수 있음
- 이메일 대신 더 안전한 채널(신뢰 연락처 등)로 교체하면 위험 감소

### 3-2. 비밀번호 파생 키 + SSS
 
비밀번호로 평소에 키를 파생하고, SSS 조각을 복구 수단으로 사용하는 방식.
비밀번호를 까먹어도 SSS 조각으로 데이터 키를 직접 복원할 수 있음.
 
**최초 가입 시**
```
비밀번호 설정
        ↓
데이터 암호화 키 생성
        ↓
이 키를 SSS로 3개 조각으로 분산
        ↓
조각1 → 서버 저장
조각2 → 이메일 전송
조각3 → 기기 보관
```
 
**평소 사용**
```
비밀번호 입력
        ↓
잠금 해제 키 생성 (메모리에만 잠깐 존재)
        ↓
서버에서 잠긴 데이터 암호화 키 가져와 복호화
        ↓
데이터 접근
```
 
**비밀번호 분실 시 복구**
```
서버 조각1 + 이메일 조각2 수집 (2개면 복구 가능)
        ↓
데이터 암호화 키 직접 복원
        ↓
새 비밀번호 설정 후 재암호화
        ↓
복구 완료
```
 
**장점**
- 비밀번호 분실 시에도 SSS로 복구 가능
- 서버는 조각 하나만 갖고 있어서 단독 복호화 불가 → 조건1 만족
**단점**
- 두 가지 방식을 모두 구현해야 해서 복잡도 증가
- 이메일 조각 분실 시 복구 수단 소멸
- 이메일 계정 해킹 시 조각 탈취 위험

### 3-3. Secure Enclave + 니모닉
 
평소엔 기기 하드웨어로 키를 보호하고, 니모닉을 복구 수단으로 사용하는 방식.
 
**최초 가입 시**
```
니모닉 12단어 생성 → 사용자에게 발급
        ↓
니모닉으로 마스터 키 파생 → Secure Enclave에 저장
```
 
**평소 사용**
```
생체인증 → Secure Enclave에서 키 꺼내 데이터 복호화
```
 
**기기 분실 시 복구**
```
니모닉 12단어 입력
        ↓
마스터 키 복원 → 새 기기 Secure Enclave에 저장 → 복구 완료
```
 
**장점**
- 평소엔 생체인증으로 가장 편한 UX
- 서버에 키 관련 정보 일절 없음
- 하드웨어 보안 + 니모닉 복구의 조합
**단점**
- 니모닉 분실 시 복구 불가
- 니모닉 보관 UX 문제

### 3-5. PIN + SSS (소셜 로그인 환경)
 
카카오처럼 별도 비밀번호가 없는 환경에서, 앱 전용 PIN을 추가 설정해 키를 파생하고 SSS로 복구하는 방식.
 
**최초 가입 시**
```
카카오 로그인
        ↓
앱 전용 PIN 설정 (4~6자리)
        ↓
카카오 고유 ID + PIN → 키 생성
        ↓
키를 SSS로 3개 조각 분산
조각1 → 서버, 조각2 → 이메일, 조각3 → 기기
```
 
**평소 사용**
```
카카오 로그인 → PIN 입력 → 카카오 ID + PIN으로 키 재생성 → 데이터 접근
```
 
**기기 분실 시 복구**
```
새 기기에서 카카오 로그인
        ↓
서버 조각 + 이메일 조각 수집
        ↓
키 복원 → 새 PIN 설정 → 복구 완료
```
 
**장점**
- 카카오 로그인 환경에서 비밀번호 없이 CSE 구현 가능
- PIN + 카카오 ID 조합으로 단순 PIN보다 보안 강화
- SSS로 기기 분실 복구 가능
**단점**
- 사용자가 PIN을 추가로 기억해야 함
- PIN이 짧아서 무차별 대입 공격에 취약 → 시도 횟수 제한 필요
- 카카오 연동이 끊기면 키 파생 불가
